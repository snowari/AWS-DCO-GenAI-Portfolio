# DCO 인프라 핵심 용어 정리

*본 문서는 데이터센터 운영(DCO) 직무 이해를 위한 교육용 용어집입니다. 실제 회사 내부 장비명, 고객명, 일련번호는 포함하지 않습니다.*

## Rack

1. 쉬운 설명: 서버, 스위치, 전원 장비 등을 층층이 넣어두는 금속 선반 같은 장비 보관 구조입니다.
2. DCO 직무 이해에서 어디에 나올 수 있는지: 장비 위치 확인, 케이블 정리, 자산 관리, 장애 발생 위치 파악에서 자주 등장합니다.
3. 학생이 자기 말로 바꿔 쓸 때 참고할 표현: “랙은 데이터센터 장비들이 정해진 위치에 설치되는 물리적 기준점이라고 이해했습니다.”
4. 주의할 점: 실제 랙 번호나 위치 정보는 내부 정보일 수 있으므로 외부 문서나 AI 프롬프트에 적지 않습니다.

## Server

1. 쉬운 설명: 데이터를 저장하거나 계산 작업을 수행하며 서비스가 동작하도록 돕는 컴퓨터 장비입니다.
2. DCO 직무 이해에서 어디에 나올 수 있는지: 하드웨어 점검, 부품 교체, 장애 확인, 자산 관리 업무에서 나올 수 있습니다.
3. 학생이 자기 말로 바꿔 쓸 때 참고할 표현: “서버는 데이터센터에서 실제 서비스와 데이터를 처리하는 핵심 장비라고 이해했습니다.”
4. 주의할 점: 실제 서버명, 고객명, 일련번호, 내부 구성 정보는 공개하지 않습니다.

## NIC

1. 쉬운 설명: 서버가 네트워크에 연결될 수 있도록 해주는 네트워크 카드입니다.
2. DCO 직무 이해에서 어디에 나올 수 있는지: 네트워크 연결 장애, Link Down, 포트 확인, 케이블 점검 상황에서 등장할 수 있습니다.
3. 학생이 자기 말로 바꿔 쓸 때 참고할 표현: “NIC는 서버가 네트워크와 통신하기 위한 연결 장치라고 이해했습니다.”
4. 주의할 점: NIC 장애를 단정하지 말고 케이블, 포트, 스위치 상태를 함께 확인해야 합니다.

## ToR Switch

1. 쉬운 설명: Top of Rack Switch의 약자로, 랙 안의 서버들을 네트워크에 연결해주는 스위치입니다.
2. DCO 직무 이해에서 어디에 나올 수 있는지: 여러 서버의 네트워크 연결 상태를 확인하거나 Link Down 원인을 점검할 때 나올 수 있습니다.
3. 학생이 자기 말로 바꿔 쓸 때 참고할 표현: “ToR Switch는 같은 랙 안의 서버들이 네트워크에 연결되도록 돕는 장비라고 이해했습니다.”
4. 주의할 점: 실제 스위치명, 포트 번호, 구성 정보는 내부 정보일 수 있으므로 그대로 공유하지 않습니다.

## Link Down

1. 쉬운 설명: 서버, 스위치, 케이블 사이의 네트워크 연결이 끊겼거나 정상 연결되지 않은 상태입니다.
2. DCO 직무 이해에서 어디에 나올 수 있는지: 네트워크 장애 확인, 케이블 연결 점검, 포트 상태 확인, Ticket 작성 상황에서 나올 수 있습니다.
3. 학생이 자기 말로 바꿔 쓸 때 참고할 표현: “Link Down은 네트워크 연결이 정상적으로 살아 있지 않은 상태라고 이해했습니다.”
4. 주의할 점: 원인을 바로 단정하지 말고 케이블, 포트, NIC, 스위치 상태를 순서대로 확인해야 합니다.

## CRC error

1. 쉬운 설명: 데이터가 이동하는 과정에서 손상이나 전송 오류가 감지된 상태입니다.
2. DCO 직무 이해에서 어디에 나올 수 있는지: 네트워크 품질 문제, 케이블 불량 의심, 광케이블 오염, 포트 이상 확인에서 나올 수 있습니다.
3. 학생이 자기 말로 바꿔 쓸 때 참고할 표현: “CRC error는 데이터 전송 중 오류가 반복되는지 확인해야 하는 신호라고 이해했습니다.”
4. 주의할 점: 한 번의 오류만으로 원인을 확정하지 말고 반복 여부와 관련 장비 상태를 함께 기록해야 합니다.

## SSD failure

1. 쉬운 설명: 서버에 장착된 SSD 저장장치가 고장 났거나 정상적으로 동작하지 않는 상태입니다.
2. DCO 직무 이해에서 어디에 나올 수 있는지: 하드웨어 장애 대응, 부품 교체, 자산 기록, Incident Report 작성에서 나올 수 있습니다.
3. 학생이 자기 말로 바꿔 쓸 때 참고할 표현: “SSD failure는 서버 저장장치에 문제가 생겨 정해진 절차에 따라 확인해야 하는 상태라고 이해했습니다.”
4. 주의할 점: 임의로 부품을 분리하거나 교체하지 않고 반드시 SOP와 담당자 지시에 따라야 합니다.

## Fan Alert

1. 쉬운 설명: 서버 내부 팬에 문제가 있거나 냉각이 정상적으로 되지 않을 가능성을 알려주는 경고입니다.
2. DCO 직무 이해에서 어디에 나올 수 있는지: 서버 상태 점검, 온도 경고 확인, 하드웨어 장애 대응에서 나올 수 있습니다.
3. 학생이 자기 말로 바꿔 쓸 때 참고할 표현: “Fan Alert는 서버 냉각 상태를 확인해야 한다는 경고라고 이해했습니다.”
4. 주의할 점: 팬 문제는 온도 상승과 연결될 수 있으므로 Temperature warning과 함께 확인해야 합니다.

## SOP

1. 쉬운 설명: Standard Operating Procedure의 약자로, 반복 업무를 정해진 순서대로 처리하기 위한 표준 작업 절차입니다.
2. DCO 직무 이해에서 어디에 나올 수 있는지: 부품 교체, 케이블 점검, 장애 대응, 자산 관리 등 반복 업무 전반에서 나올 수 있습니다.
3. 학생이 자기 말로 바꿔 쓸 때 참고할 표현: “SOP는 실수를 줄이기 위해 따라야 하는 표준 절차라고 이해했습니다.”
4. 주의할 점: 기억에 의존해 처리하지 말고 최신 절차와 담당자 지시를 확인해야 합니다.

## SLA

1. 쉬운 설명: Service Level Agreement의 약자로, 서비스나 요청을 정해진 기준과 시간 안에 처리하기 위한 약속입니다.
2. DCO 직무 이해에서 어디에 나올 수 있는지: 장애 우선순위 판단, 작업 처리 시간 관리, Escalation 필요 여부 판단에서 나올 수 있습니다.
3. 학생이 자기 말로 바꿔 쓸 때 참고할 표현: “SLA는 정해진 시간과 기준 안에 업무를 처리하기 위한 약속이라고 이해했습니다.”
4. 주의할 점: 실제 SLA 수치나 고객별 기준은 내부 정보일 수 있으므로 공개하지 않습니다.

## Escalation

1. 쉬운 설명: 스스로 해결하기 어렵거나 권한이 필요한 문제를 상위 담당자나 관련 팀에 전달하는 과정입니다.
2. DCO 직무 이해에서 어디에 나올 수 있는지: 장애 영향이 크거나 SOP만으로 해결하기 어려운 상황, SLA에 영향을 줄 수 있는 상황에서 나올 수 있습니다.
3. 학생이 자기 말로 바꿔 쓸 때 참고할 표현: “Escalation은 혼자 판단하기 어려운 문제를 필요한 담당자에게 정확히 전달하는 과정이라고 이해했습니다.”
4. 주의할 점: 감정이나 추측보다 확인된 사실, 발생 시간, 현재 상태, 시도한 조치를 중심으로 전달해야 합니다.

## Incident Report

1. 쉬운 설명: 장애나 문제 상황의 원인, 영향, 조치 내용, 결과를 정리한 보고서입니다.
2. DCO 직무 이해에서 어디에 나올 수 있는지: 장애 대응 이후 원인 정리, 재발 방지, 팀 공유 자료 작성에서 나올 수 있습니다.
3. 학생이 자기 말로 바꿔 쓸 때 참고할 표현: “Incident Report는 문제가 어떻게 발생했고 어떻게 처리되었는지 정리하는 보고서라고 이해했습니다.”
4. 주의할 점: 과장하거나 추측하지 말고 확인된 내용과 조치 결과를 구분해 작성해야 합니다.

## Handover Memo

1. 쉬운 설명: 근무 교대나 담당자 변경 시 현재 상황과 이어서 해야 할 일을 전달하는 메모입니다.
2. DCO 직무 이해에서 어디에 나올 수 있는지: 교대 근무, 진행 중인 작업 인수인계, 미해결 Ticket 공유에서 나올 수 있습니다.
3. 학생이 자기 말로 바꿔 쓸 때 참고할 표현: “Handover Memo는 다음 담당자가 업무를 이어받을 수 있도록 상황을 정리하는 메모라고 이해했습니다.”
4. 주의할 점: 누락이 생기지 않도록 현재 상태, 남은 작업, 주의사항을 분명히 적어야 합니다.

## Temperature warning

1. 쉬운 설명: 장비나 랙 주변 온도가 기준보다 높아지거나 냉각 문제가 의심될 때 발생하는 경고입니다.
2. DCO 직무 이해에서 어디에 나올 수 있는지: 서버 상태 점검, 냉각 문제 확인, Fan Alert 대응, 장애 예방 업무에서 나올 수 있습니다.
3. 학생이 자기 말로 바꿔 쓸 때 참고할 표현: “Temperature warning은 장비가 과열될 가능성이 있어 냉각 상태를 확인해야 한다는 신호라고 이해했습니다.”
4. 주의할 점: 온도 경고를 가볍게 넘기지 말고 장비 상태와 주변 환경을 함께 확인하고 필요 시 Escalation해야 합니다.

## 협업 흐름 예시

Link Down이나 CRC error 같은 경고가 발생하면 먼저 Ticket에 상황을 기록하고, SOP에 따라 케이블, 포트, NIC, ToR Switch 상태를 확인합니다. 정해진 절차만으로 해결이 어렵거나 SLA에 영향을 줄 수 있으면 Escalation을 진행합니다. 이후 조치 결과는 Incident Report나 Handover Memo에 정리하여 다음 담당자와 팀이 같은 상황을 이해할 수 있도록 공유합니다.

## 3. 헷갈리는 용어 비교

내가 비교한 용어: SOP / SLA

내가 이해한 차이: SOP는 작업을 어떤 순서로 처리할지 알려주는 표준 절차이고, SLA는 정해진 기준이나 시간 안에 처리해야 하는 약속입니다. 즉 SOP는 “어떻게 할 것인가”에 가깝고, SLA는 “언제까지 또는 어떤 기준으로 처리할 것인가”에 가깝다고 이해했습니다.

아직 헷갈리는 부분: 실제 상황에서 SLA에 영향을 줄 정도로 중요한 문제인지 판단하는 기준은 더 많은 예시를 보며 익혀야 할 것 같습니다.

내가 비교한 용어: Ticket / Incident Report

내가 이해한 차이: Ticket은 문제 상황이나 작업 요청을 처음 기록하고 처리 과정을 추적하는 문서이고, Incident Report는 장애가 끝난 뒤 원인, 영향, 조치 결과를 정리하는 보고서입니다. 즉 Ticket은 진행 중인 업무 관리에 가깝고, Incident Report는 사후 정리와 재발 방지에 가깝다고 이해했습니다.

아직 헷갈리는 부분: 어떤 문제를 단순 Ticket으로만 남기고, 어떤 문제를 Incident Report까지 작성해야 하는지는 더 학습이 필요합니다.

내가 비교한 용어: Link Down / CRC error

내가 이해한 차이: Link Down은 네트워크 연결 자체가 끊기거나 정상 연결되지 않은 상태이고, CRC error는 연결은 되어 있지만 데이터 전송 과정에서 오류가 감지되는 상태입니다. 즉 Link Down은 연결 상태 문제에 가깝고, CRC error는 전송 품질 문제에 가깝다고 이해했습니다.

아직 헷갈리는 부분: 두 문제가 동시에 발생할 때 어떤 순서로 확인해야 하는지는 SOP 예시를 통해 더 익히고 싶습니다.
